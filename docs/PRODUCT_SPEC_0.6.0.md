# 西新世紀末物語 — 0.6.0製品仕様

更新日：2026-07-14

## 1. 文書の位置づけ

この文書は、0.6.0アーリーアクセス基盤について、プロデューサーが承認した製品要件と、featureブランチ上の実装候補が採用している構成・初期値・検証基準を整理する。

0.6.0は本書更新時点では候補状態であり、`main`への反映、tag、GitHub Release、ChatGPT Sites公開、製品受入を意味しない。現在の公開状態は[PROJECT_STATE.md](PROJECT_STATE.md)、実装順は[PRODUCT_ROADMAP.md](PRODUCT_ROADMAP.md)、個別の実装・テスト・QA・レビュー結果はIssue #29と対象Draft PRを正式な記録先とする。

本書では、情報を次の四区分で扱う。

- **承認済み要件**：0.6.0で成立させる製品仕様
- **候補実装**：現在の作業ブランチでコードとして確認できる方式。受入前の候補
- **暫定値・暫定表現**：試遊、実機QA、正式アート制作で変更可能
- **将来範囲**：0.6.0では正式採用・完成させない要素

## 2. ブランド

### 承認済み要件

- 正式名称：`西新世紀末物語`
- 読み：`にしじんせいきまつものがたり`
- `アーリーアクセス版`はタイトル補助表記であり、正式名称には含めない
- `202X`表記はプレイヤー画面で使用しない
- 0.5.0以前の`ASHFALL OUTPOST`表記、repository名、既存公開URLは履歴として維持する
- `西新世紀末物語`のロゴ表示はタイトル画面に置き、通常の戦闘HUDへ常設しない
- 主要なプレイヤー向け表示は日本語を主体とし、内部識別子の`CRAWLER`、`SCOUT`、`RANGER`、`BREAKER`、`BRAWLER`、`GUNNER`、`MEDIC`をそのまま表示名に使わない
- 0.6.0で`CRAWLER`に相当するプレイヤー向け暫定表示は`移動拠点`とする。stable ID、コード上のkind、履歴文書内の固有用語は英語のままでよい

### 暫定表現

0.6.0候補のロゴはCSSによる文字主体のロゴタイプである。正式ロゴアート、パイセン以外の正式キャラクターアート、背景、会話演出は将来差し替え可能とする。

## 3. アーリーアクセスのゲームループ

### 承認済み要件

次の一連の体験を一画面ゲームとして成立させる。

**タイトル → 導入イベント → エリアマップ・ステージ詳細 → 編成 → 戦闘前イベント → 戦闘 → 星評価・報酬 → 戦闘後イベント → エリアマップへ復帰 → 次ステージ解放**

- 序章名は`新たな世界の始まり`
- ステージ順は配列位置だけに依存させず、stable ID、前提、解放条件、次の解放先を持つデータで管理する
- クリア済みステージは再プレイできる
- エリアマップのステージノードへ過去最高星を直接表示する
- 3ステージ完了後、百道浜・福岡タワー方面の異常反応を示す

### 候補実装

- タイトル、会話、エリアマップ、編成、戦闘、リザルトを明示的な画面状態で切り替える
- 会話データと表示を分離し、背景、左右立ち絵、名前、役割、表情、テキスト送り、スキップ、ログ、暗転、揺れを宣言的に指定する
- 将来地点は`通信途絶`、`調査中`、`経路封鎖`、`高危険区域`として表示し、選択不能にする
- 未開始時の主操作は`物語を始める`、開始済みセーブでは`物語を続ける`とし、通常の開始・続行で既存進行を消去しない
- `最初から始める`とマップ上のセーブ初期化は通常の開始・続行から分離し、どちらも確認ダイアログを必須とする

## 6. 星評価と報酬

### 星評価

星は勝敗と戦闘終了時のプレイヤー側移動拠点の残存HP率だけで算出する。敵側感染拠点、敵拠点、`barricadeHp`、撃破数、経過時間などを星評価へ使用しない。

| 結果 | 星 |
|---|---:|
| 敗北、未クリア、残存HP 0% | 0 |
| 勝利かつ残存HP 1%以上70%未満 | 1 |
| 勝利かつ残存HP 70%以上90%未満 | 2 |
| 勝利かつ残存HP 90%以上 | 3 |

- 保存するのは各ステージの過去最高星とし、低い再挑戦結果で下げない
- 1%、70%、90%はデータとして変更できる初期閾値であり、正式バランスの固定値ではない

### 報酬

暫定通貨名は`補給物資`とする。

| ステージ | base reward | 各星の初回到達報酬 |
|---|---:|---:|
| 西新商店街 | 100 | 50 |
| 早良区役所 | 140 | 70 |
| 西新防衛線・TAKUYA | 200 | 100 |

| 今回の星 | 通常クリア報酬倍率 |
|---|---:|
| 星1 | base reward × 1.00 |
| 星2 | base reward × 1.25 |
| 星3 | base reward × 1.50 |

- 初めて到達した星の報酬は各マイルストーンにつき一度だけ付与する
- 初回で星3なら星1・2・3の初回到達報酬をまとめて付与する
- 星1から星3へ更新した場合は、未取得の星2・3分だけ追加する
- リザルトでは今回の星、過去最高星、記録更新、通常報酬、新規星到達報酬、合計を分けて表示する
- 各戦闘開始時に一意な結果レシートIDを発行し、同じIDの星更新、通常報酬、初回星報酬、ユニット・ステージ解放、ローカル保存は一度だけ適用する
- リロード、戻る連打、リザルト再表示、同一結果の再処理後も結果レシート台帳を保持し、過去の同じ結果を再び支払対象にしない
- base reward、倍率、星到達報酬は試遊後に調整する暫定バランス値である

## 8. 戦場、標的、接敵テンポ

### 候補実装

- 960×540、3レーン、一画面戦闘、既存キャラクターの描画寸法を維持する
- 0.5.0基準の拠点間有効距離612に対し、0.6.0候補は765とし、25%拡張する
- 味方目標接触点`x=110`、敵目標接触点`x=875`、味方出撃点`x=205`、敵戦闘開始点`x=805`
- CRAWLER描画境界は`x=-112, width=310`、敵拠点は`x=850, width=184`とし、拠点の一部を画面外へ置いて戦場を確保する
- 速度、射程、出撃位置、拠点位置は共通geometry/configへ集約する
- 拡張分を無意味な徒歩時間やキャラクター縮小だけで処理しない

通常攻撃は次の制約を持つ。

- 近接は同一レーンかつ攻撃距離内だけ
- 遠距離は同一レーン優先。隣接レーンは役割設定、射程、明示的な射線成立時だけ
- 2レーン離れた通常攻撃は禁止
- 攻撃開始時に標的IDを固定し、弾道、着弾、ダメージ対象を一致させる
- 回復・支援役は味方集団から過度に離れない
- 死体は標的・移動障害・衝突対象にしない

## 9. 死亡、感染、焼却

### 敵

敵は`alive → dying → corpse → ashing → removed`で遷移する。HP 0以降は非targetable・非blockingとする。

| 区分 | dying | corpse | ashing |
|---|---:|---:|---:|
| 通常 | 0.42秒 | 4.8秒 | 1.45秒 |
| 重装 | 0.68秒 | 6.2秒 | 1.9秒 |
| ボス | 1.15秒 | 8秒 | 2.5秒 |

表示上限は全体36体、1レーン14体とし、画面外または超過した古い非戦闘状態を安全に除去する。これらの秒数と上限は実ブラウザのテンポ・性能確認で調整可能な候補値である。

### 味方

味方は`alive → dying → ally corpse → infection warning → generic zombie`で遷移する。

- ゾンビ化まで12秒
- 残り6秒から軽い警告、残り3秒から強い警告
- 上記は内部タイマーだけに保持し、戦闘画面へ秒数、数字カウントダウン、円形タイマー、進捗バーを表示しない
- 予兆は死体の痙攣、姿勢変化、皮膚・目の色調変化、煙と控えめな効果音で伝える
- 復活時は汎用ゾンビとなり、元の武器、必殺技、役割能力を継承しない
- 復活直後は0.75秒の起き上がり硬直を持つ
- 死体は非blocking
- 死亡・感染状態は戦闘ローカルであり、再戦、戦闘終了、マップ復帰で破棄する

### 火炎焼却

- 地面上のfire areaが味方死体へ接触した時点で感染タイマーを停止する
- 約2秒の燃焼後に灰となり、復活しない
- 燃焼時間の数値や進捗は表示せず、炎、煙、黒化、灰化のアニメーションだけで伝える
- 死体を直接タップする専用操作にはしない
- 火炎範囲の判定と表示を一致させ、pause中は感染・燃焼タイマーを進めない

## 11. iPhone Safariとvisual viewport

### 承認済み要件

- iPhone横画面でゲームルートを実際の表示可能領域の中央へ置き、左右どちらにも偏らせない
- 不自然な左右の黒い余白をなくし、ゲーム全体を単純縮小して解決しない
- SafariのブラウザUI表示中も、主要HUD、ユニットボタン、物資ボタンを表示可能領域内へ収める
- `100vh`固定に依存せず、safe areaとvisual viewportの変化へ追従する
- PC横画面を壊さない

### 候補実装

- viewport metaへ`viewport-fit=cover`を指定する
- `visualViewport`が使える場合は`width`、`height`、`offsetLeft`、`offsetTop`をCSS変数へ同期し、使えない場合は`window.innerWidth`／`innerHeight`へfallbackする
- `resize`、`orientationchange`、visual viewportの`resize`／`scroll`、Canvasの`ResizeObserver`で再計算する
- ルートはvisual viewportの幅・高さ・オフセットへ固定し、`100dvh`はJavaScript同期前の補助fallbackに限る
- `safe-area-inset-left/right/top/bottom`はゲーム全体を縮める外側余白ではなく、上部情報、下部操作、マップ、編成、会話など個々のUIへ内側余白として適用する
- Canvasは960×540を縦横同倍率でcoverし、中央オフセットを使う。入力座標にも同じscaleとoffsetの逆変換を使う
- 高さ390px以下と350px以下の横画面ではHUDとキャンペーン画面を再構成し、文字と操作領域を保つ
- 幅の広い短い横画面ではキャラクター縮尺を下げず、3レーン中心を短い表示領域用の配置へ切り替える。回転・resize時は出撃済みユニット、死体、戦場物、効果、出撃待ちも同じレーンへ再配置する
- 新規配置する戦場物資、その回復・燃焼範囲、航空支援の着弾判定にも現在のレーン中心を使用し、短い横画面と標準画面の往復後も表示位置と当たり判定を一致させる

### 実ブラウザで証拠を残す検証基準

この文書の記載だけをQA完了の証拠にしない。Issue #29と対象Draft PRへ、少なくとも次を画面寸法・スクリーンショット・計測値とともに記録する。

1. PC 1280×720以上での表示と操作
2. 844×390横画面での左右余白量と中央配置
3. SafariブラウザUI表示を模した低いvisual viewport（例：844×340）での上下の切れ
4. 下部の全ユニット・物資ボタンが表示され、操作できること
5. 縦向きから横向きへ回転した直後の再配置
6. Safari UI表示中と縮小後に相当する二つの高さでの比較
7. タイトル、マップ、編成、会話、戦闘、リザルトの文字切れ・重なり
8. console warning/error 0件

左右余白量、中央差、上端・下端、各ボタン矩形は実測し、単なるページ表示や844×390の静的エミュレーションだけで完了扱いしない。

## 12. 暫定・将来範囲

### 0.6.0で試遊後に調整できるもの

- 星閾値、base reward、倍率、星到達報酬

- 4〜5レーン、複数CRAWLER、チャレンジモード
- `main`反映、tag、Release、Sites公開は別のリリース承認工程
