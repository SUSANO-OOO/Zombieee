# ASHFALL OUTPOST — 企画・仕様引き継ぎ

## 1. 文書の目的と主読者

この文書は、プロデューサーとChatGPTがASHFALL OUTPOSTの作品意図、長期構想、承認済みゲーム仕様を共有し、企画・仕様判断を継続するための引き継ぎである。

ChatGPTは、企画、仕様整理、比較、ロードマップ、ストーリー、プレイヤー体験評価、Codex向け指示書、Codex報告の評価を支援する。実装や検証を行うCodexとは役割を混同しない。タスクごとのモデル指定はGitHub Issueで管理し、この恒久文書のタイトルや役割名には含めない。

高水準の役割分担は次のとおり。

- プロデューサー：作品の方向性、採否、優先順位、各工程の最終承認
- ChatGPT：企画、仕様整理、比較、ロードマップ、ストーリー、プレイヤー体験評価、Codex向け指示書、報告評価、Issue整理
- Codex：技術調査、実装、テスト、検証

Codexには、Issueごとに選ぶ二つの実行場所がある。

- Local：ローカル状態、Windows、実ブラウザ、PC横画面、844×390、タッチ、回転、音、実ゲーム進行、Sites公開前確認
- Cloud：GitHub基準の長時間・横断的作業、隔離環境、バックグラウンド処理、差分・PR候補

Codex Cloudは必須ではなく、LocalやCloudを独立した意思決定主体として扱わない。同一タスクを両方へ重複依頼せず、Cloud成果をLocalで同じように再実装しない。詳細な権限、承認工程、受け渡し、検証義務は[AGENTS.md](../AGENTS.md)を正式な所有元とする。

技術状態を判断する際は、未確認のローカル状態を推測せず、次の正式所有元を確認する。

- 開発運用規則：[AGENTS.md](../AGENTS.md)
- 公開・検証・保留状態：[PROJECT_STATE.md](PROJECT_STATE.md)
- セットアップと主要ファイル：[README.md](../README.md)
- 個別タスクの判断と報告：対応するGitHub Issue

## 2. 作品概要

ASHFALL OUTPOSTは、CRAWLERを守りながら生存者部隊を指揮し、3レーンの戦場を押し返して敵陣の鉄柵を破壊するリアルタイムのCanvas防衛ゲームである。

現行の技術構成はNext.js、React、TypeScript、Canvas、vinext/Vite、Cloudflare Sites。永続データは現時点でD1・R2ともに使用していない。変動する公開版や公開URLは[PROJECT_STATE.md](PROJECT_STATE.md)を参照する。

## 3. 作品意図と長期構想

この作品は、現在の1ステージだけで完結させる予定ではない。将来的には、次のような方向へシリーズを拡張する。

- プレイアブルキャラクター
- 新しいモンスター、エリート、ボス
- 複数ステージ
- ストーリー、世界設定、登場人物の関係
- ゲームモード、成長要素、リプレイ性
- 同時に大量の敵が出現する戦闘、敵密度や物量を活かした展開
- 開発中に生まれる新しい戦闘、演出、システム、遊び方

この一覧は候補を固定するものではない。現段階で新要素を急いで増やしていないのは、基本戦闘、スマートフォン操作、勝敗進行、検証環境を安定させ、将来の追加を安全に行える土台を作るためである。

一方で、安定化を理由に拡張を無期限に延期しない。新しいコンテンツを決める際は、先にプレイヤーへ与えたい体験を定義し、その追加に必要な最小限の構造だけを整える。将来を見越した全面リファクタリングを先行させない。

## 4. 承認済みゲーム仕様

- ゲーム世界は960×540を基準とするCanvasゲーム。
- 戦場は上・中・下の3レーンで、ユニットはY座標を使ってレーン間を移動できる。
- 味方はCRAWLERの出撃ベイから順次出撃し、出撃待ちは最大3枠。
- 敵の最終目標はCRAWLER。
- 敵はCRAWLERへ前進し、進路を物理的に塞ぐ味方だけを迎撃する。
- 防衛・均衡・攻勢の3作戦を切り替えられる。
- TAKUYAはWAVE 8で登場するボス。
- TAKUYA撃破前は鉄柵を攻撃できない。
- TAKUYA撃破後、3レーンすべてを覆う鉄柵が攻撃可能になる。
- 鉄柵は全レーン共通のHPを持つ。
- 鉄柵HPが0になると勝利。
- CRAWLER HPが0になると敗北し、CRAWLERと鉄柵が同時に0の場合も敗北を優先する。
- 6種類の味方ユニットと4種類の支援行動がある。
- 開始後に5秒間の準備時間がある。
- 一時停止、BGM、SFX、キーボード操作、タッチ操作に対応する。

新要素を追加する場合も、プロデューサーの承認なしにCRAWLER、3レーン、TAKUYA、鉄柵の基本進行を壊さない。仕様変更の比較、採否、実装範囲はGitHub Issueで管理する。

## 5. 対応プラットフォームの製品要件

この節は、対応プラットフォームに関する製品・プレイヤー体験上の方針の正式所有元である。

- 主対象のプレイ環境はスマートフォン横画面。
- PC横画面も正式対応。
- スマートフォンでは、横画面のタッチ操作を中心に、読みやすく押しやすいプレイヤー体験を提供する。縦画面の利用者には横画面への回転案内を示す。
- PCでは、横画面とキーボードで正式にプレイでき、スマートフォン向け最適化によってPC体験を損なわない。

具体的な検証環境、画面サイズ、確認条件、承認工程は[AGENTS.md](../AGENTS.md)、実施済みの検証記録は[PROJECT_STATE.md](PROJECT_STATE.md)を参照する。

## 6. ChatGPTの企画・仕様整理上の役割

ChatGPTは、プロデューサーの意図を次の順で実行可能な仕様へ整理する。

1. 目的と狙うプレイヤー体験を確認する。
2. 現行仕様、非対象、変更してはいけない要素を確認する。
3. 大きな判断が残る場合は、実装を前提にせず複数案を比較する。
4. プロデューサーが採用した案だけを承認済み仕様として整理する。
5. 個別タスクは、一つの目的に限定したGitHub IssueまたはCodex指示へ落とし込む。
6. Codexの報告を、実装の有無だけでなく狙ったゲーム体験になったかという観点で評価する。

ChatGPTは未確認のコード、Git状態、公開状態、検証結果を確認済みとして扱わない。必要な技術判断は、ローカルCodexまたはCodex Cloudの担当範囲を明示して依頼する。
