## 7. 画像承認と視覚同一性

すべての新規画像は一枚ごとの明示承認制。

1. 一度に一枚だけ提示
2. Asset ID、用途、予定パス、revision、短い意図を記載
3. `この画像で進めてよいですか`と確認
4. `承認`前に正式採用、派生制作、統合、次画像提示へ進まない
5. 修正は同Asset IDの新revision
6. 却下画像は再利用しない
7. 承認履歴をIssue #37、PR #38、単一のmachine-readable asset manifestへ記録

承認待ち中もsave、AI、UI、戦闘ロジック、テストなど画像非依存作業は並行継続する。

人物の基準アートが承認された時点で、顔型、目、眉、鼻、口、顎、年齢感、性別表現、髪、体格、衣服、主要配色、武器、装備、姿勢、シルエットをidentity lockとする。

ポートレート、立ち絵、表情差分、スプライトは、承認済み基準アートから別人化させない。派生画像は元の承認済み画像と並べて確認できる証拠を添える。

保全4名は既存採用画像をidentity lockとする。透過、crop、余白、atlas、アニメーション補正は可能だが、人物の顔、髪、体格、服、武器、主要配色を変更しない。

個別画像の承認だけで実ゲーム統合を合格にしない。実ゲームへ統合後、844×390と844×340で次を確認する。

- 同一人物に見える
- 顔、武器、役割シルエットが潰れない
- 頭、手足、武器が切れない
- 足元が論理レーンへ接地する
- UI、字幕、HP、カードと不自然に重ならない
- 背景と人物の明度・彩度が喧嘩しない
- AI量産顔、フォトリアル、プラスチック3Dへ逸脱しない

## 10. 戦力拡充と戦闘中コスト経済

友人要望の低コストヒーラーと範囲殲滅型アタッカーは、新規人物を増やさず既存11名へ割り当てる。

### ナオ：低コストヒーラー

- 配置コストの初期基準：`35`
- Codexの調整許容範囲：`32〜38`
- 低HP、低攻撃力、移動可能な単体救護を主役にする
- 最低HP比率の味方を優先して治療する
- 治療後に短時間の被ダメージ軽減を付与する
- 同じナオを複数体召喚可能
- 複数のナオが同一対象を同時治療する場合、二体目以降の回復効率を段階的に減衰させる
- `簡易救護所`は定点範囲回復、ナオは移動単体回復として差別化する
- コストを許容範囲外へ変更する場合は新たな承認が必要

### 範囲殲滅型アタッカー

- クレイジーキング：近接円形範囲。前線へ入り、密集感染者をまとめて殲滅する高リスク型
- レイダー：同一レーンの直線範囲、貫通または弾幕制圧。離れた群れを削る中遠距離型
- タタラ：範囲殲滅へ寄せず、装甲、重装敵、感染拠点の破壊へ専念
- クレイジーキングとレイダーが同じ使用感にならないよう、射程、対象形状、攻撃間隔、過熱・回転数、対単体性能を分ける
- 画面全体を無条件に消す攻撃や、特定人物なしでは突破不能な敵配置は禁止

### 戦闘中コスト上限

- 現行上限`100`から拡張する
- 初期実装基準：`150`
- Codexの調整許容範囲：`140〜160`
- 範囲外へ変更する場合は新たな承認が必要
- 上限拡張の目的は、二体の中高コストユニットまたは三体前後の低コストユニットを計画的に貯めて展開できること
- 上限を上げるだけで開始コストと自然回復速度を同時に強化しない
- 第一調整では既存の開始コストと自然回復速度を維持する
- 簡単になりすぎる場合は、開始コスト、自然回復、再出撃待ち、敵圧力を個別に調整する
- 上限、開始コスト、自然回復の三項目を同時に上方修正することは禁止
- 最終値はStage 1〜6の成功率、初回出撃時間、コスト溢れ率、連続展開回数、CRAWLER残存HPを記録して確定する

## 11. 手動戦術と味方AI

前進、後退、防御、標準、突撃など、同目的の手動戦術機能はUI、入力、説明、状態、保存項目ごと撤去する。

ユニット配置コスト、再出撃待ち、支援ゲージなど別用途の仕組みは残す。

味方AI：

- 攻撃可能な敵を無視して前進しない
- 接触中、通行阻害、CRAWLER攻撃中の敵を優先
- hitbox重複中に前進しない
- TAKUYA撃破後も残存・後続敵を再索敵
- Yずれ、レーン移動、target消失後に次tickで再取得
- 実ブラウザで敵素通り再現0件

敵素通り、非現実的な接触、表示とhitboxの不一致は公開阻害条件。

- スマートフォンが主対象
- 公開前に844×390、844×340、Playwright WebKitのiPhone相当profile、touch、safe area、回転を必須検証
- 利用可能な物理iPhoneまたは実機クラウドがある場合は公開前または公開後に確認する
- 物理iPhone・実機クラウドが利用できない場合、844×390、844×340、Playwright WebKitのiPhone相当profile、touch、safe area、回転、画面ロック・復帰、タブ復帰の全成功を正式代替ゲートとし、公開・Issue close・branch削除まで進めてよい
- 代替ゲートを使用した場合は、物理実機未確認であることと実施した代替証拠を最終報告へ明記する
- 物理iPhone確認を実施し重大不具合があった場合はIssueを閉じず、通常のrevert PRで復旧する
- PCは基本回帰確認
- schema v5
- v2〜v4 migration
- localStorageとIndexedDBを独立検証
- 一方破損時は正常な方から復旧
- 両方破損時は黙って初期化しない
- 手動バックアップ、破損データ書出し、完全初期化
- 星、報酬、加入の二重適用禁止
