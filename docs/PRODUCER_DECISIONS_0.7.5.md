# Part A — Version 0.7.1

## 4. Stage 5 hotfix

### 4.1 不採用仕様

Stage 5「西新駅・ホーム／線路区域」の必須台車護衛方式は不採用とする。

理由：

- 台車が敵の進行方向へ自ら進む
- 敵接近で停止し、敵数・汚染に応じて継続ダメージを受ける
- 味方AIが台車追従へ引かれ、迎撃と噛み合わない
- 実プレイで進行不能級
- 本作の基本戦闘「CRAWLERを守り、敵拠点を破壊する」と整合しない
- 面白さより理不尽さと進行阻害が上回る

### 4.2 固定する修正方向

- Stage 5のstable IDを変更しない
- save、星、報酬、既読、Stage 6解放を維持
- 必須の移動台車耐久を勝敗条件から外す
- ホーム制圧・感染拠点破壊中心の通常戦闘へ変更
- 台車は背景演出、戦闘後搬送、または敗北条件ではない副目標としてのみ残せる
- 特定ユニット、調達ユニット、最大強化を必須にしない
- 3種類以上の異なる編成で通常クリア可能にする
- 0.7.5のレーン撤廃、戦闘基盤、育成、画像刷新をhotfixへ先取りしない

## 5. Stage 6ボス撃破不能hotfix

### 5.1 確認された現象

Stage 6ボス「改札喰い」のHPが504まで減った後、それ以上減らず、撃破できない。

### 5.2 固定する修正方向

- 公開版または同等baselineで現象を再現する
- HP表示、damage計算、target／hit判定、無敵状態、phase遷移、state同期、boss objective、勝利判定を切り分ける
- `504`という数値だけを特別扱いする応急処置は禁止
- 根本原因を修正する
- phase閾値が意図された仕様なら、正しい遷移後に再びダメージ可能にする
- 複数のユニット・武器構成でHPが504未満へ減り、0まで到達することを確認
- ボス撃破、勝利判定、結果画面、報酬適用、マップ帰還が一度だけ成立する
- 敗北、再挑戦、撤退、再読込を確認する
- 公開版でもボスを実際に撃破する

## 6. 0.7.1互換性

- stable unit ID、stable stage IDを維持
- 既存saveを自動初期化しない
- 所有、発見、調達、編成、星、キャップ、解放、既読、設定、戦闘ボイス設定を維持
- 0.7.1のために0.7.0 tag、Release、履歴を変更しない

目的：

## 8. 長期方向

- 本編ステージ数は50を基準とし、固定上限にしない
- プレイアブルユニットは30体を基準とし、将来追加可能にする
- Challenge Modeは本編と別枠で、1セッション3ステージ、約6セッションを将来目標とする
- 1.0までは戦闘中心で進める
- 長尺ストーリーイベントの量産は1.0までの主対象にしない

- 全ユニットを本編ストーリーへ登場させる必要はない
- 敵勢力は将来、人間敵を含めて拡張可能にする

## 9. 量産基盤

Codexは現行リポジトリを監査し、フォルダ構成、データ形式、schema、registry、loader、generator、validator、migration方式を決定する。

完成状態：

- runtimeとcontentの責任が分離される
- 新ユニット追加で既存runtimeを広範囲に編集しない
- 新敵追加で敵別・地域別の巨大mechanicsファイルを増殖させない
- 新ステージ追加でcampaign本体へ大量の個別コードを書かない
- stable IDと表示名を分離
- aliasとsave migrationを維持
- 正本、旧版、自動生成物、QA証拠、未使用候補を区別
- 未使用候補は参照検索、build、test、実ブラウザQA後にのみ削除

最低限データ駆動化する対象：

- units、enemies、stages、missions、waves／spawn
- maps／battle spaces、difficulty、rewards／unlock
- acquisition、upgrades、battle events
- side／challenge／future timed events
- assets、portraits、sprites、animation、VFX、audio
- save／migration

最低限自動化する対象：

- 新規コンテンツ雛形生成
- schema検証
- ID重複・参照切れ・アセット不足検出
- 未使用候補検出
- balance simulation
- caps economy simulation
- save migration matrix
- synthetic scale test

合成試験だけで完成としない。productionへ配信しないfixtureとして、generatorからテスト用ユニット1体、敵1体、ステージ1件を作成し、定型登録、決定的再生成、dry-run、build、QA gallery、fixture除去後差分0まで確認する。

## 10. 戦場と固定3レーン

プレイヤーから見える固定3レーン制と、contentの固定lane番号依存を撤廃する。

- 戦場は連続した空間として見える
- ユニット、支援物、設置物を固定3レーンへ強制しない
- 攻撃、射程、回復、範囲効果、衝突、能力は座標、距離、形状、範囲、視線を基準にする
- 設置予告、可否、理由、必要に応じた最寄り有効位置補正を持つ

内部性能のため、経路ノード、navigation mesh、flow field、steering、見えない移動帯、経路グラフは使用可能。ただしプレイヤーへ固定レーンを強制しない。

## 11. プレイフィールド、CRAWLER、敵拠点

プレイフィールド：

- スマートフォンでも狭く見えない構図
- CRAWLERと敵拠点が戦闘領域を過剰に占有しない
- キャラクターを豆粒化しない
- 戦場、カメラ、拠点、出現位置、HUD、縮尺を一体で再設計
- 原則として一画面で戦況を把握

CRAWLER：

- 世界観に合う装甲移動司令拠点へ再デザイン
- 明確なドア、ハッチ、スロープ
- 味方は実際の出口から出撃
- 縦線からワープする表現を撤去
- ドア、照明、足音、走り出しを同期
- 配置禁止は本体と最小安全域に限定
- 車両横・前方へ支援物を置ける

敵拠点：

- 細く、スマートで視認しやすい
- 戦場を過剰に隠さない
- HP、被弾、破損、崩壊が理解できる
- 敵は拠点後方、内部、画面外から自然に出現

## 12. キャラクター画像といくらちゃん

各キャラクターに用途別profileを持たせる。

- identity master
- event portrait
- formation card
- battle sprite
- 必要ならencyclopedia

crop、focus point、scale、anchor、safe areaを用途別に定義できる。

品質：

- イベント全身押し込みによる見切れ0
- 編成カードの顔位置、余白、倍率を統一
- PC、844×390、844×340で顔欠け、武器欠け、極端な全身縮小、UI干渉0
- 足元、頭頂、体格、武器、影、hitbox、選択判定を標準化
- 小柄、標準、大柄、重装の自然な体格差を残す
- タタラを大柄な人物として自然に見せる

いくらちゃん：

- 18歳の成人女性
- 可愛さ、親しみやすさ、案内役としての視認性を維持
- 現行r7以上の露出度を維持し、露出量を減らさない
- 胸元、肩、腹部、太ももなど現在の露出方向を保つ
- 衣服面積を増やして世界観へ合わせない
- 世界観への適合は、色調、素材、タクティカル装備、通信端末、汚れ、擦れ、補修跡、使用感で行う
- 通信・地図・情報分析担当として成立させる

## 13. Animation、VFX、AI

固定セル・固定状態前提を、可変フレームanimation clipへ移行する。

最低状態：idle、move、wind-up、active、recovery、hit、incapacitated、death、専用行動。

各clipで、フレーム時間、loop、攻撃判定、投射物位置、武器先端、SE、VFX、移動可否、硬直、向き、接地、body scaleを設定可能にする。

武器別profile：素手、鈍器、チェーンソー、拳銃、ライフル、狙撃銃、マシンガン、クロスボウ、設置武器、回復・支援。

- 単純な十字・線・×印を正式な武器軌道、衝撃、接触、hit stop、敵反応へ置換
- マシンガンは構え、反動、銃口発光、連射音、弾道、薬莢、複数ダメージ、着弾、被弾反応を同期

味方・敵AIを行動profile化し、接敵、適正距離、危険敵優先、負傷者優先、CRAWLER防護、目的物防護、迂回、再索敵、詰まり解消、味方間隔を扱う。主要脅威には複数の対応方法を用意し、特定ユニット必須を避ける。

## 14. 調達、強化、難易度

編成画面へ購入を混在させず、人員管理画面に次を設ける。

- 所有一覧
- 調達
- 強化

購入と強化にはキャップを使用する。

0.7.5の育成はminimum viable progression vertical sliceに限定する。

- 全11ユニットが共通基盤から強化可能
- 数段階の成長
- 役割を壊さない少数の節目効果
- 後発ユニットの追いつき措置
- stable IDで保存
- migrationと経済simulation
- 誤投資で本編進行不能にしない
- 育成だけで押し切れず編成判断も必要
- 最大強化、特定ユニット、過剰周回を本編必須にしない

大規模スキルツリー、多数の固有能力、装備在庫、レアリティ、ガチャ、覚醒、限界突破、複雑なrespecは対象外。

## 15. イベント基盤

通常戦闘と同じ部品から、戦闘中イベント、サイド任務、高難易度作戦、Challenge Mode、将来の期間イベントを登録できるようにする。

イベント専用の別戦闘エンジンを作らない。0.7.5では大型期間イベント、新規大量アート、長尺ストーリーを制作せず、登録、表示、開始、終了、保存、復刻の差し込み口までを対象とする。

## 16. スマートフォン性能

スマートフォン横画面を第一基準とする。

- 844×390
- 844×340
- WebKit iPhone相当
- PC基本回帰

0.7.1公開版と同一QA環境でbaselineを取得し、0.7.5本実装前にperformance budgetをIssue #44へ記録する。

第一目標：

- 60Hz相当環境で通常戦闘median 50fps以上
- p95 frame time 33ms以下
- warm-up後から15分終了時までmedian frame time悪化10%以内
- memory指標増加25%以内
- 100ms超long taskが連続しない
- background中はsimulation、描画、不要audio更新を停止
- 復帰後にgame loop、BGM、SE、voiceを二重化しない
- browser自動reload、進行消失、入力不能0

物理iPhoneを利用できない場合、発熱確認済みと断定しない。

## 17. セーブと音声

維持対象：stable unit／stage ID、所有、発見、調達、強化、編成、星、キャップ、解放、既読、設定、戦闘ボイス設定。

必要ならschemaを更新し、migration前snapshot、last-known-good、localStorage、IndexedDB、片側破損・両側破損復旧、export／importを維持する。旧saveを自動初期化しない。

音声：

- ストーリー会話の全文読み上げ、フルボイスは実装しない
- 出撃、攻撃、被弾、戦闘不能の人間キャラクター戦闘ボイスは維持
- 武器音、身体反応音、敵ボイスを削除しない
- background復帰後のBGM、SE、voice二重再生を防ぐ
