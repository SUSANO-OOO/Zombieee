## 5. 固定済み物語条件

- 店舗の正式表記：**大衆居酒屋「くまや」**
- 主人公はプレイヤー。
- 発生前のプレイヤーは大衆居酒屋「くまや」の常連客。
- 発生夜、プレイヤーは避難経路を示して客の退避をまとめる。
- その後、CRAWLERの発見・確保・運用判断の中心となり、発生から四十三日後には指揮官となる。
- プレイヤーは基本的に無言。
- プレイヤー名入力なし。UI表示は`指揮官`。
- 固定台詞、性格選択、会話分岐、好感度、別エンディングなし。
- ユウタは登場させない。
- プロローグに戦闘、操作チュートリアル、追加ミッションを入れない。
- 最初の戦闘は既存Stage 1。
- 研究所・研究員の会話シーンを導入に置かない。
- UIで伝えられる数値や条件を人物へ朗読させない。
- 通常会話は背景＋立ち絵＋下部テキストを基本とする。
- 重要場面は必要に応じて全画面一枚絵または漫画的カットを使う。

script versionは`outbreak-origin-v7`を基礎とする。ただし本書で変更された名称、加入、音響、アート担当を反映して`outbreak-origin-v8`へ現在化してよい。

## 7. キャラクターデザイン基準

- 日本の商業2Dゲーム、青年漫画、ダークアクション漫画を基礎にする。
- フォトリアル、実写写真、映画用リアルコンセプトアート、プラスチック質感の3Dレンダーにしない。
- 幼い萌えキャラ、ちびキャラ、過度なデフォルメにしない。
- 明確な輪郭線、読みやすい色面、手描き感を持たせる。
- スマートフォンで縮小しても役割と人物を判別できる。
- プレイヤーが一覧で「使いたい」と感じる魅力を持たせる。
- 全員を同じAI美男美女の顔へ揃えない。

人物ごとに、顔型、目、眉、鼻、口、顎、年齢感、髪型、髪色、肌、身長、肩幅、体格、姿勢、主配色、武器、装備の輪郭を明確に変える。

禁止：

- 不自然に完全対称な顔
- 全員同じ小顔・同じ鼻・同じ目
- 陶器のような肌
- 意味のないベルトやポーチ
- 全員黒い戦術服
- 役割と関係ない武器
- 別キャラクターの顔・身体・手・武器の流用
- 色替えだけの別人化
- フレームごとに別人になるスプライト
- 仮シルエットや単色図形の正式採用

役割の視覚化：

- ハチ：高機動、経路確保、対高速
- ミズチ：精密射撃、特殊個体優先
- 医療支援：回復、救助、損害抑制
- タタラ：破砕、装甲・拠点破壊
- レイダー：中遠距離、連射、制圧
- ガンテツ：大型盾、防御、味方保護
- モンキー：工兵、罠、設備、足止め
- ノイズ：通信、地図、情報分析。非戦闘

## 8. 戦闘スプライト

- 制作基準方向は右向き。
- 待機、移動A、移動B、攻撃予備、攻撃、被弾を最低限持つ。
- 死亡を被弾と共用して品質が落ちる場合は別状態にする。
- 6セル入力を派生atlasへ変換する方式、または7状態×左右2方向atlasのどちらでもよい。
- 全フレームで同じ顔、髪、服、体格、武器、配色を維持する。
- 足元の接地位置、頭の高さ、縮尺を統一する。
- セル間補間漏れを防ぐ透明gutterを設ける。
- 武器、手足、装備を切らない。
- 非対称装備が単純反転で破綻する場合は左右専用フレームを用意する。
- manifest、content rect、anchor、hitboxは実画像から計測する。

## 11. 敵・ボス

追加対象：

- 絡手：長い腕で拘束・引き寄せ
- 漏泥：床汚染で移動・配置へ圧力
- 走鬼：高速で前線を抜け後方侵入
- 改札喰い：駅設備と研究容器が感染組織で一体化した大型特殊個体

共通基準：

- 人間由来の感染体であることを残す。
- 福岡、西新、地下鉄、病院の世界観と結びつける。
- ファンタジー悪魔、宇宙怪獣、既存ゾンビの単純色替えにしない。
- 能力がシルエットと予備動作から理解できる。
- グロテスクさだけを目的にしない。
- スマートフォンの縮小画面でも種類を判別できる。
- 制作基準方向は右向き。
- 初見殺しではなく、対処可能な攻撃予備動作を持たせる。

改札喰いは西新駅でしか成立しない固有ボスとする。

## 14. 数値・バランス

ユニット、敵、ステージ、キャップ価格、報酬、前進・後退命令の初期値はCodexが決定する。

条件：

- 確定値、理由、許容調整幅を製品仕様へ記録する。

## 15. Safariセーブ消失・破損対策

PR #39で`main`へ入った次を削除しない。

- localStorage書込後の読み戻し確認
- IndexedDBへの自動二重保存
- localStorage消失時のIndexedDB復元
- 両方へ保存できない場合の開始・破壊的操作停止
- 保存不能警告
- 保存関連テスト

0.7.0ではさらに、localStorageとIndexedDBを両方読み、両方を独立検証する。

必要挙動：

1. 両方の保存候補を読む。
2. JSON解析、schema、必須フィールド、型、内部整合性を検証する。
3. v2〜v4をv5へmigration候補化する。
4. 有効な候補のうちrevisionまたは更新日時が新しい方を採用する。
5. 一方が破損していれば正常な方から復旧する。
6. 復旧したことを短く表示する。
7. 両方破損なら黙って初期化しない。
8. 手動バックアップ読込、破損データ書出し、完全初期化を選べる回復画面を出す。
9. migration前saveと直前正常saveを退避する。
10. migration失敗時に旧saveを上書きしない。

schema v5には少なくとも次を持たせる。

- schemaVersion
- revision
- updatedAt
- 完全性確認値
- campaignStarted
- completedStageIds
- stars
- caps
- ownership/discovery/recruitable state
- formation presets
- selected preset
- story state
- settings

必須テスト：

- localStorageのみ正常
- IndexedDBのみ正常
- localStorage破損＋IndexedDB正常
- IndexedDB破損＋localStorage正常
- 両方破損
- 両方消失
- 保存拒否・読み戻し不一致
- v2/v3/v4→v5
- migration途中失敗
- 結果・報酬直後の再読込
- 日をまたいだ再起動相当
- Safari通常タブ・プライベート相当
- バックアップ書出し・読込
- 完全初期化とキャンセル
- 星、報酬、加入の二重適用防止
