# ASHFALL OUTPOST 開発運用ルール

## 1. 適用範囲

この文書は、このリポジトリで作業するローカルCodex、Codex Cloud、および作業を指示・確認する人に適用する恒久的な運用ルールである。個別タスクの目的、許可範囲、禁止事項、完了条件、承認段階、停止条件はGitHub Issueまたはプロデューサーの明示指示で定義する。個別指示とこの文書が衝突し、対象範囲を安全に確定できない場合は、変更せずプロデューサーへ確認する。

## 2. 正しい情報源

情報は次の所有元を使い分ける。

- 正式な開発コード：プロデューサーが承認したGitHub `main`上のコミット
- 作業中コード：ローカルまたはCodex Cloudの作業ブランチ上にある候補状態
- 公開版：ChatGPT Sitesで実際に稼働している内容と、その公開基準コミット
- ゲーム仕様と長期方針：承認済みの[企画・仕様引き継ぎ](docs/CHATGPT_HANDOFF.md)、および対象範囲が明確なプロデューサーの最新の明示的な決定・承認
- 製品の実装順・依存関係・完了条件：[製品ロードマップ](docs/PRODUCT_ROADMAP.md)
- 現在の公開・検証・保留状態：[プロジェクト状態](docs/PROJECT_STATE.md)
- 起動、テスト、Lint、ビルド手順：[README](README.md)と`package.json`
- Windows、ブラウザ、実機固有の挙動：対象環境で行った最新の検証結果
- 個別タスクの議論、比較、報告、判断：対応するGitHub Issue

プロデューサーの最新の明示的な決定・承認は、その対象範囲内で有効とする。相談、仮説、未確定案、検討中の発言は、自動的に正式決定へ昇格させない。

既存の正式文書と新しい明示決定が衝突する場合は、新しい明示決定を対象範囲内で優先する。長期的な正式仕様として次のセッションでも利用する決定は、Issueの完了要約または該当する所有文書へ記録する。確定した製品判断をチャット内だけに残して終わらせない。文書更新は、対象ファイルと目的を含む実装ミッションまたはリリース承認の範囲内で行う。

Issue内の案やCodexの提案は、プロデューサーが明示的に承認するまでは正式仕様ではない。

GitHub `main`の現在のHEADは、GitHub API／GitHub画面または`git ls-remote origin refs/heads/main`でリモートrefを直接確認する。`origin/main`を使う場合は、先に`git fetch origin`などで更新済みであることを確認する。更新していない`origin/main`を現在のGitHub `main`として扱わず、文書へ「最新SHA」として複製しない。

## 3. 役割、実行場所と決定権

- プロデューサー：方向性、採否、優先順位、および各承認工程の最終判断
- ChatGPT：企画、仕様整理、比較、ロードマップ、ストーリー、プレイヤー体験評価、Codex指示書、報告評価、Issue整理
- Codex：技術調査、実装、テスト、検証。IssueごとにLocalまたはCloudを実行場所として選ぶ
- Local：ローカル状態、Windows、実ブラウザ、PC横画面、844×390、タッチ、回転、音、実ゲーム進行、Cloud成果とSites公開前の最終確認
- Cloud：GitHub基準の長時間・横断的作業、隔離環境での調査・実装、差分・PR候補作成

Codex Cloudは必須担当ではなく、独立した意思決定主体でもない。利用するかどうかは各Issueで選択する。

### 3.1 ChatGPTの判断責任

ChatGPTは、プロデューサーの案を無条件に肯定・追認しない。決定事項として明示されたものを除き、発言は検討材料として扱う。

各提案を、少なくとも次の観点から評価する。

- 事実か、仮説か、好み・方針か
- 現行仕様・過去の決定と矛盾しないか
- 依存関係と実装順が妥当か
- より速い、安い、安全、面白い代案がないか
- 将来の手戻り、性能、保守、プレイヤー体験への副作用がないか
- Codexへ渡す前に、製品全体の目的と整合しているか

必要に応じて「妥当」「一部修正が必要」「反対」「根拠不足」「技術確認待ち」を明示する。賛成する場合も検討理由を示し、反対または修正が必要な場合は理由と代案を提示する。最終採否はプロデューサーが行うが、ChatGPTは分析責任と推奨案の提示を放棄しない。未確認のコード、Git状態、公開状態、検証結果を確認済みとして扱わない。

### 3.2 Codexが決定しない事項

Codexは、次を独自に決定または正式採用しない。必要な分析と代案は示すが、最終判断はプロデューサーが行う。

- 製品方針、機能採否、優先順位、ストーリー、正式アート方向、ゲーム名称
- 難易度・バランスの大幅変更、ロードマップ変更
- 課金、DLC、有料外部サービス、ライセンス不明素材
- 認証方式、秘密情報、本番データの破壊的変更
- 正式採用、`main`反映、リリース、公開

branch protection、repository ruleset、secrets、認証、課金、本番データなどの管理操作は、実装ミッションやリリース承認へ暗黙に含めない。対象と操作を特定した専用の明示承認を必要とする。

## 4. 二段階承認方式

開発運用は、実装ミッションとリリース承認の二段階で扱う。各承認は、明示された対象と操作だけに有効であり、個別ミッションがより狭い停止条件を定めた場合はそれに従う。

### 4.1 実装ミッション

プロデューサーが目的、対象、非対象、変更禁止事項、基準refまたはSHA、作業ブランチ、完了条件、検証、停止条件を一度明示承認した後、Codexはその範囲内で次を連続実行できる。

- 現状調査、技術設計、サブエージェント委任
- featureブランチまたはworktreeの作成
- 承認範囲のコード、テスト、文書、対象アセットの編集
- 自動テスト、Lint、型検査、本番ビルド
- 必要なPC横画面、844×390、ブラウザ確認
- 問題修正と再検証の反復
- 通常commit
- 指定featureブランチへの通常push
- `main`向けDraft PRの作成と更新
- 自己レビュー、Issue報告

停止地点は、検証結果と未確認事項を備えたマージ候補のDraft PR、または個別ミッションで指定された地点とする。実装ミッションは、Draft PRのReady化、マージ、`main`変更、tag、Release、Sites公開、Issueクローズ、ブランチ削除を許可しない。

### 4.2 リリース承認

リリース承認は、少なくともPR番号、baseブランチとbase SHA、headブランチとhead SHA、merge方式、許可する操作を固定した別の明示承認とする。承認後にbaseまたはheadが移動した場合、条件が一致しない場合、検証が失敗した場合は、それ以降を実行せず停止し、再承認を待つ。

リリース承認に明示して含めた場合に限り、次を連続実行できる。

- PR最終確認とDraft PRのReady化
- PRを通した`main`への通常マージ
- マージ後の実際のresult SHAの取得と照合
- result SHAを指すannotated tagとGitHub Release
- 同じ基準SHAのChatGPT Sites公開と公開URL確認
- Issue最終要約とクローズ
- マージ済みブランチの安全な削除
- リリース用bundleの作成、verify、ref照合、SHA-256記録

tag、Release、Sites、正式bundleの基準は、マージ前のhead SHAではなく、マージ後に確認したresult SHAとする。一つでも条件不一致または失敗があれば、残りの操作を成功扱いせず停止する。

### 4.3 操作の境界

- `main`はPRを通してのみ変更し、直接pushしない
- featureブランチへの通常pushとDraft PR作成は、実装ミッションに両方が明記されている場合に連続実行できる
- リリース承認に含まれないtag、Release、Sites、Issue、ブランチ操作は行わない
- 完了後は差分、commit、push、PR、CI、検証、未確認事項、次に必要な承認を報告する

## 5. ローカルCodexとCodex Cloud

- Codexの実行場所はIssueごとにLocalまたはCloudから選ぶ。Cloudの利用は必須ではない
- 一つの目的をローカルCodexとCodex Cloudへ重複依頼しない
- Issueに担当側、基準コミットまたはref、作業ブランチ、成果物、検証担当を明記する
- Cloud成果をLocalで同じように再実装しない
- Localは取得、ビルド、テスト、Windows、実ブラウザ、PC横画面、844×390、タッチ、回転、音、実ゲーム進行、Sites公開前確認など、環境依存の最終確認を担う
- 受け渡し時は、基準コミット、変更範囲、未確認事項、必要なローカル検証を同じIssueへ残す

### 5.1 サブエージェント

- サブエージェントは個別ミッションの承認範囲を拡大できない
- 読み取り監査や独立レビューを優先して委任し、書き込み、Git操作、最終統合はメインエージェントが管理する
- 複数エージェントが同じファイルを無調整で並行編集しない
- サブエージェントの結果も現在のコード、GitHub、実環境と照合し、未確認事項を成功扱いしない

## 6. GitHub Issue運用

- 一目的一Issueとし、同じ目的の補足は同じIssueで続ける
- 目的が変わる場合は新しいIssueを使う
- Issue本文に、目的、承認段階、担当、対象・非対象、基準SHA、作業ブランチ、禁止事項、停止条件、完了条件、必要な検証を記載する
- 重複Issueは削除せず、正規Issueと重複理由を報告する。リンク、コメント、クローズは許可された操作だけを行う
- Draft PRにはbase、head、head SHA、変更範囲、検証結果、未確認事項を記録する
- Issueを閉じる前に、目的、実施結果、変更ファイル、コミット・PR・`main`・公開の各状態、未完了事項、後続Issueを要約する。Issueクローズはリリース承認または専用の明示承認がある場合だけ行う
- ロードマップ本文を個別Issueへ複製せず、正式所有元へリンクする

## 7. 製品方針に基づく検証義務

スマートフォン横画面を主対象とし、PC横画面も正式対応とする製品方針の正式所有元は[企画・仕様引き継ぎ](docs/CHATGPT_HANDOFF.md)である。この章は、その方針に基づく作業上の検証義務を定める。

- 960×540基準のCanvasゲームとして、基本確認にはPC横画面とスマートフォン相当844×390の両方を含める
- スマートフォン側では、タッチ操作、UIの可読性と押しやすさ、縦画面時の回転案内を確認する
- PC側では、キーボード操作、PC表示、スマートフォン向け変更による回帰を確認する
- タッチ、回転案内、音、端末性能、大量描画に影響する変更では、必要に応じて実機スマートフォン確認を提案する
- 調査または文書のみの変更では、不要なゲームテストを自動的に実行せず、未実施であることを報告する
- 公開成功と製品受入完了を分離する。公開基盤が成功しても、完了条件を実機で満たさない場合はIssueを完了扱いにしない

## 8. 変更時の安全原則

- 作業開始時にGit状態、対象ファイル、対応Issueを確認する
- 既存の未コミット変更を勝手に消去、復元、上書きしない
- 過去の会話や古い報告だけから現在のコード・公開・検証状態を推測しない
- 調査、設計、レビューだけを依頼された場合は実装しない
- 指定された変更禁止事項を守り、必要な権限や情報がなければ無理な代替操作をしない
- 現行のCRAWLER、3レーン、TAKUYA、敵拠点の基本進行を、承認なしに変更しない
- 将来の可能性だけを理由に全面リファクタリングや過剰な抽象化を先行させない
- プレイヤー体験単位でまとめるが、検証不能な巨大差分にはしない

### 8.1 恒久禁止

- `main`への直接push
- force push
- 共有済み履歴のrebaseまたはamend
- 無承認のSites公開、Issueクローズ、branch protectionまたはruleset変更
- 秘密情報のcommit、ライセンス不明素材の追加
- 作業範囲外ファイルの無断変更
- 未確認事項や失敗を成功扱いすること
- サブエージェントによる権限範囲の拡大
- 複数エージェントによる同一ファイルへの無調整な並行書き込み
