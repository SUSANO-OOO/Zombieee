# ASHFALL OUTPOST 開発運用ルール

## 1. 適用範囲

この文書は、このリポジトリで作業するローカルCodex、Codex Cloud、および作業を指示・確認する人に適用する恒久的な運用ルールである。個別タスクの目的、許可範囲、禁止事項、完了条件はGitHub Issueで定義し、この文書と衝突する場合はプロデューサーへ確認する。

## 2. 正しい情報源

- 正式な開発コード：プロデューサーが承認したGitHub `main`上のコミット
- 作業中コード：ローカルまたはCodex Cloudの作業ブランチ上にある候補状態
- 公開版：ChatGPT Sitesで実際に稼働している内容と、その公開基準コミット
- ゲーム仕様と長期方針：[docs/CHATGPT_HANDOFF.md](docs/CHATGPT_HANDOFF.md)
- 製品の実装順・依存関係・完了条件：[docs/PRODUCT_ROADMAP.md](docs/PRODUCT_ROADMAP.md)
- 現在の公開・検証・保留状態：[docs/PROJECT_STATE.md](docs/PROJECT_STATE.md)
- 起動、テスト、Lint、ビルド手順：[README.md](README.md)と`package.json`
- 個別タスクの議論、比較、報告、判断：対応するGitHub Issue

相談、仮説、未確定案、検討中の発言は、自動的に正式決定へ昇格させない。長期的に利用する確定事項は、Issueの完了要約または該当する所有文書へ記録する。確定した製品判断をチャット内だけに残して終わらせない。

GitHub `main`の現在のHEADは、GitHub API／GitHub画面または`git ls-remote origin refs/heads/main`でリモートrefを直接確認する。更新していない`origin/main`を現在のGitHub `main`として扱わない。

## 3. 役割と決定権

- プロデューサー：方向性、採否、優先順位、各工程の最終判断
- ChatGPT：企画、仕様整理、比較、ロードマップ、ストーリー、プレイヤー体験評価、Codex指示書、報告評価、Issue整理
- Codex：技術調査、実装、テスト、検証
- Local：Windows、実ブラウザ、PC横画面、844×390、タッチ、回転、音、実ゲーム進行、Sites公開前確認
- Cloud：GitHub基準の長時間・横断的作業、隔離環境、差分・PR候補作成

### 3.1 ChatGPTの判断責任

ChatGPTはプロデューサーの案を無条件に肯定・追認しない。決定事項以外は検討材料として扱い、少なくとも次を評価する。

- 事実、仮説、好み・方針の区別
- 現行仕様・過去決定との矛盾
- 依存関係と実装順
- より速い、安い、安全、面白い代案
- 将来の手戻り、性能、保守、プレイヤー体験への副作用
- Codexへ渡す前の製品全体との整合性

必要に応じて「妥当」「一部修正が必要」「反対」「根拠不足」「技術確認待ち」を明示する。賛成時も理由を示し、反対時は代案を出す。最終採否はプロデューサーが行うが、ChatGPTは分析責任と推奨案の提示を放棄しない。

## 4. 独立した承認工程

1. 調査
2. 実装
3. コミット
4. pushまたはPR作成
5. `main`反映
6. ChatGPT Sites公開
7. Issueクローズ
8. ブランチ削除

一つの承認は次工程の承認を含まない。依頼された工程だけを実行し、完了後は差分、検証結果、未確認事項、次に必要な承認を報告する。

## 5. LocalとCloud

- 同じ目的をLocalとCloudへ重複依頼しない
- Issueに担当、基準ref、作業ブランチ、成果物、検証担当を明記する
- Cloud成果をLocalで再実装しない
- 受け渡し時は、基準コミット、変更範囲、未確認事項、必要なローカル検証を同じIssueへ残す

## 6. GitHub Issue運用

- 一目的一Issue
- 目的が変わる場合は新しいIssue
- Issue本文に目的、承認工程、担当、基準、禁止事項、完了条件を記載
- 重複Issueは正規Issueへリンクしてクローズ
- Issueを閉じる前に、目的、結果、変更ファイル、コミット・PR・`main`・公開状態、未完了事項、後続Issueを要約
- ロードマップ本文を個別Issueへ複製せず、正式所有元へリンク

## 7. 検証義務

- 960×540基準
- PC横画面と844×390を基本確認
- スマートフォンではタッチ、可読性、押しやすさ、回転案内
- PCではキーボード操作と表示回帰
- タッチ、音、端末性能、大量描画へ影響する変更は必要に応じて実機確認
- 調査・文書のみの変更で不要なゲームテストを実行しない
- 公開成功と製品受入完了を分離する

## 8. 安全原則

- 作業開始時にGit状態、対象ファイル、対応Issueを確認
- 未コミット変更を勝手に消去・上書きしない
- 古い報告だけから現在状態を推測しない
- 調査・設計だけの依頼では実装しない
- 承認なしにCRAWLER、3レーン、TAKUYA、敵拠点の基本進行を変更しない
- 将来の可能性だけを理由に全面リファクタリングや過剰抽象化を先行させない
- プレイヤー体験単位でまとめるが、検証不能な巨大差分にはしない

## 9. 文書の所有関係

- `README.md`：入口、セットアップ、文書案内
- `AGENTS.md`：情報源、役割、承認工程、判断・検証原則
- `docs/CHATGPT_HANDOFF.md`：作品意図、長期構想、承認済み仕様
- `docs/PRODUCT_ROADMAP.md`：実装順、依存関係、バージョン目的、完了条件
- `docs/PROJECT_STATE.md`：公開版、公開基準、検証記録、未完了作業、停止地点
- GitHub Issue：個別タスクの議論、比較、報告、承認履歴

同じ変動値や規則本文を複数文書へ複製しない。
