# ASHFALL OUTPOST 開発運用ルール

## 1. 適用範囲

この文書は、このリポジトリで作業するローカルCodex、Codex Cloud、および作業を指示・確認する人に適用する恒久的な運用ルールである。個別タスクの目的、許可範囲、禁止事項、完了条件はGitHub Issueで定義し、この文書と衝突する場合はプロデューサーへ確認する。

## 2. 正しい情報源

情報は次の所有元を使い分ける。

- 正式な開発コード：プロデューサーが承認したGitHub `main`上のコミット
- 作業中コード：ローカルまたはCodex Cloudの作業ブランチ上にある候補状態
- 公開版：ChatGPT Sitesで実際に稼働している内容と、その公開基準コミット
- ゲーム仕様と長期方針：承認済みの[企画・仕様引き継ぎ](docs/CHATGPT_HANDOFF.md)、および対象範囲が明確なプロデューサーの最新の明示的な決定・承認
- 製品の実装順・依存関係・完了条件：[製品ロードマップ](docs/PRODUCT_ROADMAP.md)
- 現在の公開・検証・保留状態：[プロジェクト状態](docs/PROJECT_STATE.md)
- 起動、テスト、Lint、ビルド手順：[README](README.md)と`package.json`
- Windows、ブラウザ、実機固有の挙動：対象環境で行った最新の検証結果
- 個別タスクの議論、比較、報告、判断：対応するGitHub Issue

プロデューサーの最新の明示的な決定・承認は、その対象範囲内で有効とする。相談、仮説、未確定案、検討中の発言は、自動的に正式決定へ昇格させない。

既存の正式文書と新しい明示決定が衝突する場合は、新しい明示決定を対象範囲内で優先する。長期的な正式仕様として次のセッションでも利用する決定は、Issueの完了要約または該当する所有文書へ記録する。確定した製品判断をチャット内だけに残して終わらせない。文書更新自体は独立工程として承認を得る。

Issue内の案やCodexの提案は、プロデューサーが明示的に承認するまでは正式仕様ではない。

GitHub `main`の現在のHEADは、GitHub API／GitHub画面または`git ls-remote origin refs/heads/main`でリモートrefを直接確認する。`origin/main`を使う場合は、先に`git fetch origin`などで更新済みであることを確認する。更新していない`origin/main`を現在のGitHub `main`として扱わず、文書へ「最新SHA」として複製しない。

## 3. 役割、実行場所と決定権

- プロデューサー：方向性、採否、優先順位、および各承認工程の最終判断
- ChatGPT：企画、仕様整理、比較、ロードマップ、ストーリー、プレイヤー体験評価、Codex指示書、報告評価、Issue整理
- Codex：技術調査、実装、テスト、検証。IssueごとにLocalまたはCloudを実行場所として選ぶ
- Local：ローカル状態、Windows、実ブラウザ、PC横画面、844×390、タッチ、回転、音、実ゲーム進行、Cloud成果とSites公開前の最終確認
- Cloud：GitHub基準の長時間・横断的作業、隔離環境での調査・実装、差分・PR候補作成

Codex Cloudは必須担当ではなく、独立した意思決定主体でもない。利用するかどうかは各Issueで選択する。

### 3.1 ChatGPTの判断責任

ChatGPTは、プロデューサーの案を無条件に肯定・追認しない。決定事項として明示されたものを除き、発言は検討材料として扱う。

各提案を、少なくとも次の観点から評価する。

- 事実か、仮説か、好み・方針か
- 現行仕様・過去の決定と矛盾しないか
- 依存関係と実装順が妥当か
- より速い、安い、安全、面白い代案がないか
- 将来の手戻り、性能、保守、プレイヤー体験への副作用がないか
- Codexへ渡す前に、製品全体の目的と整合しているか

必要に応じて「妥当」「一部修正が必要」「反対」「根拠不足」「技術確認待ち」を明示する。賛成する場合も検討理由を示し、反対または修正が必要な場合は理由と代案を提示する。最終採否はプロデューサーが行うが、ChatGPTは分析責任と推奨案の提示を放棄しない。未確認のコード、Git状態、公開状態、検証結果を確認済みとして扱わない。

### 3.2 Codex送信前プレフライト監査

ChatGPTは、プロデューサーへ「Codexへ送ってよい」「この指示を送ってください」と案内する前に、対象Issueと現在状態に対する統合プレフライト監査を一度だけ完了させる。監査未完了、未確認、矛盾、権限不足、プロデューサー判断待ちが一つでも残る間は、最終送信文を提示しない。

#### 発火条件

次の場合にプレフライト監査を行う。

- 新しいCodex実行指示を最終提示する直前
- 既存指示へ仕様、許可範囲、対象ファイル、アセット、依存関係、検証条件を追加・変更した後
- 前回の監査合格後に、`main`のSHA、Issue本文・コメント、作業ブランチ・PR、承認範囲のいずれかが変わった場合

相談、比較、アイデア出し、暫定案の段階では、最終送信文を出さず、必要以上のプレフライトを繰り返さない。

#### 必須確認

一回の統合監査で、対象に関係する次を確認する。

1. GitHubリモート上の現在の既定ブランチと`main` HEAD
2. 対象Issueのstate、本文、全追加コメントを時系列で確認し、競合時の優先順位が明確であること
3. `AGENTS.md`、仕様所有文書、ロードマップ、現在状態文書との整合。ただし古い`PROJECT_STATE.md`だけを現在状態の根拠にしない
4. 対象作業ブランチ、同名リモートブランチ、関連open PRの有無
5. 許可操作と禁止操作が、調査、編集、アセット、文書、依存関係、commit、push、PR、`main`、Sites、Issue、ブランチ削除まで明示されていること
6. 変更対象ファイル、実装単位、依存関係、停止条件、復旧方法が実行可能な粒度であること
7. テスト、Lint、ビルド、実ブラウザ、端末、音、ライセンスなど、対象変更に必要な検証方法が実在するコマンドと環境に合っていること
8. Codexへ判断させてはいけない未決定の製品判断、文章、台詞、デザイン、購入、ライセンス条件が残っていないこと。残る場合は、プロデューサー判断を先に得るか、差し替え可能な基盤だけへ範囲を限定すること
9. 古いIssueコメント、参考例、撤回済み案が実装指示として誤採用されないこと
10. 最終送信文がIssueの目的、許可範囲、禁止事項、完了条件を拡大・欠落・矛盾させていないこと

ローカルの未コミット変更など、ChatGPT側から直接確認できない項目は、Codexの開始時確認と停止条件へ明示する。未確認なのに確認済みとは記録しない。

#### 判定

監査結果は次のいずれかとする。

- `PASS`：Codexへ送信可能
- `CORRECTION_REQUIRED`：Issueまたは指示の修正が必要
- `PRODUCER_DECISION_REQUIRED`：プロデューサー判断が必要
- `TECHNICAL_VERIFICATION_REQUIRED`：追加の読み取り専用技術確認が必要

`PASS`以外では最終送信文を提示しない。修正が必要な場合は、送信文を先に出してから追記するのではなく、Issueまたは正式文書を修正し、同じプレフライトを完了させてから一度だけ最終送信文を提示する。

#### 記録

`PASS`時は対象Issueへ、`Codex handoff preflight: PASS`という見出しのコメントを一件だけ残す。少なくとも次を記録する。

- 確認日時
- 確認した`main` SHA
- 対象Issueと最終指示コメント
- 作業ブランチ・関連PRの状態
- 許可範囲と禁止範囲
- 未解決事項がないこと、またはCodex開始時確認へ委ねた項目
- 再監査が必要になる変更条件

#### 効率原則

- プレフライトは細切れに何度も行わず、一回の統合確認で完了させる
- 対象と無関係なファイル、Issue、履歴を網羅的に再調査しない
- 直前に確認済みで、その後変更されていない事実は再利用してよい
- `PASS`後は、発火条件に該当する変更がない限り同じ監査を繰り返さない
- ユーザーから再確認を求められた場合も、まず変更トリガーだけを確認し、変化がなければ既存の`PASS`を再利用する
- GitHubや対象環境を確認できない場合は`PASS`にしない

## 4. 独立した承認工程

次の工程は独立して扱う。一つの承認は次工程の承認を含まない。

1. 調査
2. 実装
3. コミット
4. pushまたはPR作成
5. `main`反映
6. ChatGPT Sites公開
7. Issueクローズ
8. ブランチ削除

pushとPR作成は同じ承認区分に属する別操作である。pushの承認はPR作成を含まず、PR作成の承認はpushを含まない。両方を行う場合は両方の明示承認を必要とし、Issueまたは明示承認で指定された操作だけを実行する。

依頼された工程だけを実行し、完了後は差分、検証結果、未確認事項、次に必要な承認を報告する。

## 5. ローカルCodexとCodex Cloud

- Codexの実行場所はIssueごとにLocalまたはCloudから選ぶ。Cloudの利用は必須ではない
- 一つの目的をローカルCodexとCodex Cloudへ重複依頼しない
- Issueに担当側、基準コミットまたはref、作業ブランチ、成果物、検証担当を明記する
- Cloud成果をLocalで同じように再実装しない
- Localは取得、ビルド、テスト、Windows、実ブラウザ、PC横画面、844×390、タッチ、回転、音、実ゲーム進行、Sites公開前確認など、環境依存の最終確認を担う
- 受け渡し時は、基準コミット、変更範囲、未確認事項、必要なローカル検証を同じIssueへ残す

## 6. GitHub Issue運用

- 一目的一Issueとし、同じ目的の補足は同じIssueで続ける
- 目的が変わる場合は新しいIssueを使う
- Issue本文に、目的、現在の承認工程、担当、基準、禁止事項、完了条件を記載する
- 重複Issueは削除せず、正規Issueへリンクし、理由を要約してクローズする
- Issueを閉じる前に、目的、実施結果、変更ファイル、コミット・PR・`main`・公開の各状態、未完了事項、後続Issueを要約する
- ロードマップ本文を個別Issueへ複製せず、正式所有元へリンクする

## 7. 製品方針に基づく検証義務

スマートフォン横画面を主対象とし、PC横画面も正式対応とする製品方針の正式所有元は[企画・仕様引き継ぎ](docs/CHATGPT_HANDOFF.md)である。この章は、その方針に基づく作業上の検証義務を定める。

- 960×540基準のCanvasゲームとして、基本確認にはPC横画面とスマートフォン相当844×390の両方を含める
- スマートフォン側では、タッチ操作、UIの可読性と押しやすさ、縦画面時の回転案内を確認する
- PC側では、キーボード操作、PC表示、スマートフォン向け変更による回帰を確認する
- タッチ、回転案内、音、端末性能、大量描画に影響する変更では、必要に応じて実機スマートフォン確認を提案する
- 調査または文書のみの変更では、不要なゲームテストを自動的に実行せず、未実施であることを報告する
- 公開成功と製品受入完了を分離する。公開基盤が成功しても、完了条件を実機で満たさない場合はIssueを完了扱いにしない

## 8. 変更時の安全原則

- 作業開始時にGit状態、対象ファイル、対応Issueを確認する
- 既存の未コミット変更を勝手に消去、復元、上書きしない
- 過去の会話や古い報告だけから現在のコード・公開・検証状態を推測しない
- 調査、設計、レビューだけを依頼された場合は実装しない
- 指定された変更禁止事項を守り、必要な権限や情報がなければ無理な代替操作をしない
- 現行のCRAWLER、3レーン、TAKUYA、敵拠点の基本進行を、承認なしに変更しない
- 将来の可能性だけを理由に全面リファクタリングや過剰な抽象化を先行させない
- プレイヤー体験単位でまとめるが、検証不能な巨大差分にはしない
- GitHubへの書き込み前に、対象リポジトリ、IssueまたはPR、ファイルパス、対象ブランチ、操作種別を明示的に確定する
- ファイル作成・更新・削除では、直接`main`変更が明示承認されていない限り対象ブランチを省略しない
- 書き込みAPIを機能確認、存在確認、スキーマ確認のために試行しない。確認には読み取り操作またはツール定義を使う
